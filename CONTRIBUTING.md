# 기여 가이드

SSM Server Connect 프로젝트에 기여해주셔서 감사합니다! 이 문서는 프로젝트에 기여하는 방법을 안내합니다.

## 기여 방법

### 1. 이슈 리포트

버그를 발견하거나 새로운 기능을 제안하고 싶다면:

1. [GitHub Issues](https://github.com/[username]/ssm-server-connect/issues)에서 기존 이슈를 확인
2. 중복되지 않는다면 새 이슈 생성
3. 이슈 템플릿을 사용하여 상세한 정보 제공

### 2. 코드 기여

#### 개발 환경 설정

```bash
# 저장소 포크 및 클론
git clone https://github.com/[your-username]/ssm-server-connect.git
cd ssm-server-connect

# 의존성 확인
./install.sh

# 개발용 브랜치 생성
git checkout -b feature/your-feature-name
```

#### 코딩 스타일

- **Bash 스타일**: [Google Shell Style Guide](https://google.github.io/styleguide/shellguide.html) 준수
- **함수명**: snake_case 사용 (예: `check_requirements`)
- **변수명**: 대문자 사용 (예: `AWS_PROFILE`)
- **들여쓰기**: 4칸 공백 사용
- **주석**: 복잡한 로직에는 반드시 주석 추가

#### 코드 품질

```bash
# 스크립트 문법 검사
shellcheck ssm-exec-fzf.sh

# 실행 권한 확인
chmod +x ssm-exec-fzf.sh

# 기본 테스트
./ssm-exec-fzf.sh --help
```

### 3. Pull Request 프로세스

1. **브랜치 생성**: `feature/`, `bugfix/`, `docs/` 접두사 사용
2. **커밋 메시지**: 명확하고 설명적인 메시지 작성
3. **테스트**: 변경사항이 기존 기능을 깨뜨리지 않는지 확인
4. **문서 업데이트**: 필요시 README.md 업데이트
5. **PR 생성**: 템플릿을 사용하여 상세한 설명 제공

#### 커밋 메시지 규칙

```
type(scope): 간단한 설명

상세한 설명 (선택사항)

Fixes #123
```

**타입**:
- `feat`: 새로운 기능
- `fix`: 버그 수정
- `docs`: 문서 변경
- `style`: 코드 스타일 변경
- `refactor`: 코드 리팩토링
- `test`: 테스트 추가/수정
- `chore`: 빌드 프로세스나 도구 변경

**예시**:
```
feat(auth): AWS SSO 자동 로그인 기능 추가

사용자가 SSO 세션이 만료되었을 때 자동으로 
재로그인을 시도하는 기능을 추가했습니다.

Fixes #42
```

## 개발 가이드라인

### 새로운 기능 추가

1. **요구사항 분석**: 기능의 필요성과 사용자 가치 검토
2. **설계 문서**: 복잡한 기능은 설계 문서 작성
3. **단계적 구현**: 작은 단위로 나누어 구현
4. **테스트**: 다양한 환경에서 테스트
5. **문서화**: 사용법과 예제 추가

### 버그 수정

1. **재현**: 버그를 재현할 수 있는 최소한의 케이스 작성
2. **원인 분석**: 근본 원인 파악
3. **수정**: 최소한의 변경으로 수정
4. **테스트**: 수정이 다른 부분에 영향을 주지 않는지 확인
5. **회귀 방지**: 동일한 버그가 재발하지 않도록 예방책 마련

### 테스트 가이드라인

#### 수동 테스트 체크리스트

- [ ] 다양한 AWS 프로파일에서 동작
- [ ] 여러 리전에서 인스턴스 조회
- [ ] SSO 로그인/로그아웃 시나리오
- [ ] 네트워크 오류 상황 처리
- [ ] 권한 부족 시나리오
- [ ] 빈 인스턴스 목록 처리
- [ ] FZF 인터페이스 네비게이션
- [ ] 세션 연결 및 종료

#### 테스트 환경

- **macOS**: 최신 버전 및 이전 버전
- **Linux**: Ubuntu, Amazon Linux, CentOS
- **AWS 환경**: 다양한 리전 및 계정
- **네트워크**: 다양한 네트워크 환경

## 코드 리뷰

### 리뷰어 가이드라인

- **기능성**: 요구사항을 올바르게 구현했는가?
- **안전성**: 보안 취약점이나 위험한 코드는 없는가?
- **성능**: 불필요한 API 호출이나 비효율적인 코드는 없는가?
- **가독성**: 코드가 이해하기 쉽고 유지보수 가능한가?
- **일관성**: 프로젝트의 코딩 스타일을 따르고 있는가?

### 리뷰 받는 사람 가이드라인

- **설명**: PR 설명에 변경사항과 이유를 명확히 기술
- **테스트**: 충분한 테스트를 거쳤음을 증명
- **응답**: 리뷰 피드백에 신속하고 정중하게 응답
- **수정**: 요청된 변경사항을 적극적으로 반영

## 릴리스 프로세스

### 버전 관리

- **시맨틱 버저닝**: MAJOR.MINOR.PATCH 형식 사용
- **MAJOR**: 호환성을 깨뜨리는 변경
- **MINOR**: 새로운 기능 추가 (하위 호환)
- **PATCH**: 버그 수정

### 릴리스 체크리스트
